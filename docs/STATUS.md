## Security fix — coupon once-per-customer identity (2026-07-12)

- **Found by an adversarial security review of the unpushed batch.** Pre-fix, `placeOrder` re-checked the coupon and stored the `CouponRedemption` under the SUBMITTED checkout-form email, while `loadCheckoutCart` checked logged-in carts against the account email (`accountEmail()`). A logged-in customer could therefore reuse an UNCAPPED percent coupon indefinitely by typing a fresh contact email each checkout — the account's own once-per-customer slot was never consumed (revenue leak; bounded only when `maxRedemptions` was set).
- **Fix** (`src/lib/actions/checkout.ts`): a single `couponIdentityEmail` = session email for logged-in users, submitted email for guests, used for BOTH the placeOrder recheck and the stored redemption row — so the dedup identity matches `loadCheckoutCart` and is bound to the account (the session email can't be varied via the form). Guests are byte-identical to before (no session → form email).
- Wire-verified on the prod build: logged-in order 1 with a throwaway contact email now records the redemption under the account email; a second order with a DIFFERENT throwaway email is rejected and the coupon stripped (no second redemption). The rest of the review came back clean (path traversal on the upload route, `$queryRaw` in stock.ts, iyzico amount cross-check, reservation locking, and authz on all new admin/account surfaces all verified safe).

## Step 8 architecture notes (landed 2026-07-04)

- **Route groups**: storefront chrome (header/footer/cart drawer) moved to `src/app/(storefront)/layout.tsx`; root layout is now bare (html/body/fonts/skip link); `/admin` has its own sidebar layout. Writes invalidate storefront caches with `updateTag` (read-your-own-writes in server actions; the iyzico callback keeps `revalidateTag(..., "max")` since `updateTag` is action-only): tags → `"tags"`+`"catalog"`; products/sales/inventory/orders(paid) → `"catalog"`; settings → `"settings"`.

Saves send a JSON payload; the server reconciles **by id** — kept variants are updated in place (cart items and order-item links survive price/stock edits), removed combinations are deleted (cart lines cascade; order items SetNull, snapshots intact), new combinations are created. A stale-structure check rejects saves whose ids no longer match the DB. After save the action REDIRECTS (list, or edit page on create) because client-side keys are stale post-save — do not change to an in-place re-render without syncing ids back.
